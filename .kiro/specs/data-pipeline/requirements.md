# Requirements Document

## Introduction

The Data Pipeline feature provides a synchronized multi-modal data collection system for autonomous driving machine learning. The system captures multi-camera images (Front RGB + 4 AVM cameras) paired with vehicle state data from the CARLA simulator, generating high-quality datasets with edge-case scenarios for model training. The system operates in a Windows Host (CARLA Server 0.9.15) + WSL2 Linux (Python Client) environment, utilizing NVIDIA RTX 3090 Ti / 4090 workstations.

The camera configuration consists of 5 cameras:
- Front RGB (800×600, FOV 90°): Primary input for BC/RL driving control models
- AVM Front/Rear/Left/Right (400×300, FOV 120°): Wide-angle cameras for BEV stitching and Free-Space detection

## Glossary

- **Data_Pipeline**: The complete system responsible for collecting, synchronizing, and storing multi-modal driving data from CARLA simulator
- **CARLA_Server**: The CARLA simulator instance (version 0.9.15) running on Windows Host
- **Python_Client**: The data collection client application running in WSL2 Linux environment
- **Camera_Image**: RGB image captured from the vehicle's front-facing camera at 800x600 resolution
- **AVM_Camera_Image**: Wide-angle image captured from one of the 4 AVM cameras (front/rear/left/right) at 400x300 resolution
- **BEV_Image**: Bird's Eye View image generated by stitching 4 AVM_Camera_Images via homography transformation
- **Vehicle_State**: A data record containing Speed, Steering, Throttle, and Brake values
- **Timestamp**: Millisecond-precision time value used for synchronization
- **Collection_Cycle**: A single iteration of data capture occurring at 10Hz frequency (every 100ms)
- **Driving_Log**: CSV file containing mappings between image filenames and their corresponding sensor values
- **Output_Session_Dir**: Timestamped output directory in format `src/data/{YYYY-MM-DD_HHMMSS}/`
- **Edge_Case_Scenario**: Driving conditions with varied weather (rain, fog) or time of day (night, backlight)
- **Frame_Drop**: Loss of data during a Collection_Cycle due to system performance issues
- **Headless_Mode**: Execution mode without graphical display output

## Requirements

### Requirement 1: Multi-Modal Data Capture

**User Story:** As a machine learning engineer, I want to capture synchronized multi-camera images and vehicle state data, so that I can train models with temporally aligned multi-modal inputs.

#### Acceptance Criteria

1. WHEN a Collection_Cycle occurs, THE Data_Pipeline SHALL capture one Camera_Image from the front-facing RGB camera on CARLA_Server
2. WHEN a Collection_Cycle occurs, THE Data_Pipeline SHALL capture four AVM_Camera_Images from the front/rear/left/right AVM cameras on CARLA_Server
3. WHEN a Collection_Cycle occurs, THE Data_Pipeline SHALL capture one Vehicle_State from CARLA_Server
4. THE Data_Pipeline SHALL assign identical Timestamp values to Camera_Image, all AVM_Camera_Images, and Vehicle_State captured in the same Collection_Cycle
5. THE Data_Pipeline SHALL use millisecond precision for all Timestamp values
6. THE Camera_Image SHALL have a resolution of 800x600 pixels
7. EACH AVM_Camera_Image SHALL have a resolution of 400x300 pixels
8. THE Vehicle_State SHALL contain Speed, Steering, Throttle, and Brake values

### Requirement 2: Collection Frequency

**User Story:** As a machine learning engineer, I want data collected at a minimum frequency, so that I have sufficient temporal resolution for training.

#### Acceptance Criteria

1. THE Data_Pipeline SHALL execute Collection_Cycle at a minimum frequency of 10Hz
2. WHEN operating for 1 second, THE Data_Pipeline SHALL complete at least 10 Collection_Cycle iterations
3. THE Data_Pipeline SHALL maintain consistent timing between Collection_Cycle iterations within 10ms tolerance

### Requirement 3: Image Storage

**User Story:** As a machine learning engineer, I want images stored in a standardized multi-camera directory structure, so that I can easily load them for training.

#### Acceptance Criteria

1. WHEN a Camera_Image is captured, THE Data_Pipeline SHALL save it to disk in PNG format
2. THE Data_Pipeline SHALL store Camera_Image files in the directory path `{Output_Session_Dir}/front/`
3. THE Data_Pipeline SHALL store AVM_Camera_Images in directories `{Output_Session_Dir}/avm_front/`, `{Output_Session_Dir}/avm_rear/`, `{Output_Session_Dir}/avm_left/`, `{Output_Session_Dir}/avm_right/`
4. THE Data_Pipeline SHALL name each image file using the pattern `{timestamp}.png` where timestamp is the Timestamp value in milliseconds
5. THE Data_Pipeline SHALL create all image subdirectories if they do not exist
6. THE Data_Pipeline SHALL create a new Output_Session_Dir with format `src/data/{YYYY-MM-DD_HHMMSS}/` for each collection session

### Requirement 4: Label Storage

**User Story:** As a machine learning engineer, I want vehicle state data stored in a structured format, so that I can pair it with images during training.

#### Acceptance Criteria

1. THE Data_Pipeline SHALL store all Vehicle_State records in a Driving_Log file
2. THE Driving_Log SHALL be stored at the file path `{Output_Session_Dir}/labels/driving_log.csv`
3. THE Driving_Log SHALL use CSV format with columns for image filename, Speed, Steering, Throttle, and Brake
4. WHEN a Vehicle_State is captured, THE Data_Pipeline SHALL append one row to the Driving_Log mapping the Camera_Image filename to the Vehicle_State values
5. THE Data_Pipeline SHALL create the `{Output_Session_Dir}/labels/` directory if it does not exist

### Requirement 5: Edge-Case Scenario Generation

**User Story:** As a machine learning engineer, I want data collected under varied conditions, so that my models can handle diverse real-world scenarios.

#### Acceptance Criteria

1. THE Data_Pipeline SHALL use CARLA Python API to modify simulation weather conditions
2. THE Data_Pipeline SHALL randomly vary weather between clear, rain, and fog conditions
3. THE Data_Pipeline SHALL use CARLA Python API to modify simulation time of day
4. THE Data_Pipeline SHALL randomly vary time of day between daytime, night, and backlight conditions
5. WHEN starting a data collection session, THE Data_Pipeline SHALL apply at least one weather variation
6. WHEN starting a data collection session, THE Data_Pipeline SHALL apply at least one time of day variation

### Requirement 6: Long-Duration Collection

**User Story:** As a machine learning engineer, I want to collect large datasets over extended periods, so that I have sufficient training data.

#### Acceptance Criteria

1. WHEN operating in Headless_Mode in WSL environment, THE Data_Pipeline SHALL collect data continuously for at least 1 hour
2. WHEN operating for 1 hour, THE Data_Pipeline SHALL complete at least 36,000 Collection_Cycle iterations
3. THE Data_Pipeline SHALL write all captured data to disk during collection
4. WHEN operating for 1 hour, THE Data_Pipeline SHALL experience zero Frame_Drop events

### Requirement 7: Network Communication

**User Story:** As a system operator, I want the Python client to communicate with the CARLA server, so that data collection can occur across the Windows-WSL boundary.

#### Acceptance Criteria

1. THE Python_Client SHALL connect to CARLA_Server using TCP protocol
2. THE Python_Client SHALL connect to CARLA_Server on port 2000
3. WHEN CARLA_Server is unreachable, THE Python_Client SHALL return a connection error message
4. THE Python_Client SHALL maintain the TCP connection throughout the data collection session

### Requirement 8: Headless Operation

**User Story:** As a system operator, I want to run data collection without a display, so that I can maximize system resources for data capture.

#### Acceptance Criteria

1. WHERE Headless_Mode is enabled, THE Data_Pipeline SHALL execute without requiring graphical display output
2. WHERE Headless_Mode is enabled, THE Data_Pipeline SHALL log collection progress to console or file
3. WHERE Headless_Mode is enabled, THE Data_Pipeline SHALL complete all data capture operations

### Requirement 9: Fault Tolerance and Data Preservation

**User Story:** As a machine learning engineer, I want collected data preserved even when the CARLA server crashes, so that I don't lose hours of data collection work.

#### Acceptance Criteria

1. WHEN CARLA_Server crashes or becomes unreachable during collection, THE Data_Pipeline SHALL detect the connection loss
2. WHEN connection loss is detected, THE Data_Pipeline SHALL flush all remaining data in the queue to disk
3. WHEN connection loss is detected, THE Data_Pipeline SHALL close the Driving_Log file properly
4. WHEN connection loss is detected, THE Data_Pipeline SHALL log the number of frames successfully saved before termination
5. THE Data_Pipeline SHALL preserve all data collected up to the point of failure
6. WHEN restarting after a crash, THE Data_Pipeline SHALL NOT overwrite existing data files

### Requirement 10: Multi-Camera Sensor Configuration

**User Story:** As a machine learning engineer, I want 5 cameras (1 front RGB + 4 AVM) attached to the ego vehicle with correct positions and FOV, so that I can collect both driving control data and surround-view data simultaneously.

#### Acceptance Criteria

1. THE Data_Pipeline SHALL attach a front-facing RGB camera at position (x=1.5, z=2.4) with 90° FOV and 800×600 resolution
2. THE Data_Pipeline SHALL attach 4 AVM cameras with 120° FOV and 400×300 resolution at the following positions:
   - AVM Front: (x=2.0, z=0.5), pitch=-90° (downward-facing)
   - AVM Rear: (x=-2.0, z=0.5), pitch=-90°, yaw=180°
   - AVM Left: (y=-1.0, z=0.5), pitch=-90°, yaw=-90°
   - AVM Right: (y=1.0, z=0.5), pitch=-90°, yaw=90°
3. ALL 5 cameras SHALL be synchronized within the same Collection_Cycle via CARLA synchronous mode
4. THE Data_Pipeline SHALL store each camera's images in its designated subdirectory (front/, avm_front/, avm_rear/, avm_left/, avm_right/)
5. WHEN any AVM camera fails to deliver an image within a Collection_Cycle, THE Data_Pipeline SHALL log a warning and continue collection with available cameras
